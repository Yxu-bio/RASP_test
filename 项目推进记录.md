## 1. 项目目标与路线调整

最初的目标不是把旧版 VB.NET RASP 逐行翻译成 Python，而是复刻旧版 RASP 的功能、参数入口、运行流程和结果语义。

随着 S-DIVA、S-DEC、S-BGB 的排查推进，我们确认一个关键原则：旧版 RASP 的很多结果不是 VB.NET 自己算出来的，而是由外部引擎完成。Python 版如果自行重写统计逻辑，即使界面看起来接近，也容易与旧版输出系统性不一致。

因此后续路线调整为：

- Python 负责配置窗口、输入文件生成、外部引擎调用、结果解析和展示。
- S-DIVA 尽量把真实算法交还给修改版 `DIVA.exe`。
- S-BGB 尽量按旧版方式批量调用 R/BioGeoBEARS，减少重复加载。
- S-DEC 因当前使用 `lagrange-ng.exe`，不能假装等价于旧版导师修改版 Lagrange，只对齐能可靠对齐的流程、配置和显示。

这个路线调整是整个阶段最重要的推进。它避免继续在 Python 端修补一个本来就不应由 Python 复刻的 S-DIVA 汇总算法。

## 3. S-DIVA 推进记录

### 3.1 原始问题

最早的 Python 版 S-DIVA 做法是：Python 逐棵树调用 DIVA，然后由 Python 自己把结果汇总。这导致两个问题：

- 输出结果与旧版 RASP 差距很大。
- 某些约束、Fossil、range exclude、随机树抽样等行为无法与旧版一致。

你指出旧版 RASP 的 S-DIVA 是修改版 `DIVA.exe` 自己完成 S-DIVA 统计，而不是 VB.NET 端手动汇总。这一判断后来成为 S-DIVA 改造的核心依据。

### 3.2 主要代码改动

涉及文件主要包括：

- `domain/models/sdiva_config.py`
- `gui/dialogs/sdiva_config_dialog.py`
- `application/services/sdiva_analysis_service.py`
- `application/services/tree_collection_prepare_service.py`
- `gui/main_window.py`
- 相关 worker、结果解析和可视化路径也做过配套调整。

主要改动如下：

- 删除或弱化菜单栏中独立的 `SDIVA 配置` 入口，改为点击运行 S-DIVA 时弹出配置窗口。
- S-DIVA 不再继续依赖 Python 自己的逐树汇总逻辑，而是生成旧版风格 proc/config 输入，调用修改版 `engines/diva/DIVA.exe`。
- Python 端主要负责写输入、传参数、管理子进程、解析输出。
- 增加或完善 S-DIVA 配置窗口：
  - Range constraints。
  - Include/Exclude range 列表。
  - Max areas。
  - Allow Reconstruction。
  - Allow Extinction。
  - Random Step。
  - Max reconstructions。
  - Use Final tree。
  - Fossil 设置。
  - Threads。
  - Operation 菜单：Refresh the Range List、Save Setting、Load Setting。
- `启用随机树抽样` 改为默认开启。
- `Allow Reconstruction(Slow)` 和 `Allow Extinction(Slow)` 改为默认关闭。
- 旧版默认 `keep=65536`，Python 版内部对齐；后来你确认旧版界面看不到 keep，所以配置界面中不显示 keep。
- Random Step 行为按旧版方式调整：
  - 未启用 Random Step 时，使用固定较大的搜索保留量。
  - 启用 Random Step 时，按旧版思路降低搜索步长。
- Include/Exclude range 的运行逻辑改为写入 DIVA proc，而不是仅在 Python 解析后过滤。
- 移动 range 到 Exclude 后，会像旧版一样自动勾选 Allow Extinction。
- Use Final tree 的可用性按旧版逻辑判断，只有参考树或最终树结构满足要求时启用。
- Fossil 逻辑按旧版 S-DIVA 对齐：未勾选 Use Final tree 时，Fossil 不真正约束节点；勾选 Use Final tree 后才作为最终树约束参与。
- 对 D 区域整列取消、range 全排除等情况做了兼容，避免旧版能快速完成而 Python 版直接报错。
- 针对子进程长时间不退出的问题，调整了 DIVA 调用、proc 写入和超时/日志处理方向。

### 3.3 修改原因

S-DIVA 的核心不是普通 DIVA 单树结果的 Python 后处理。旧版 RASP 使用的是修改版 DIVA，它支持 S-DIVA config/proc 模式，内部完成跨树统计。之前 Python 版自行汇总，本质上是在实现另一个算法，所以结果差距大是结构性问题，不是简单修几个百分比能解决。

因此，S-DIVA 的正确方向是让 `DIVA.exe` 负责真正的 S-DIVA 计算，Python 只做工程层适配。

### 3.4 当前状态

S-DIVA 已经从“Python 自己统计”推进到“调用修改版 DIVA 的旧版式流程”。配置界面、Fossil、Random Step、Threads、Include/Exclude、keep 默认值、Use Final tree 等已经按旧版逻辑大幅对齐。

仍需注意：旧版 `DIVA.exe` 的内部算法和输出格式是黑盒，Python 端能做的是正确写入配置和解析结果。若未来发现某个 proc 参数旧版写法不同，还需要继续根据旧版代码或实际输出修正。

你指出 S-BGB 与旧版结果对不齐时，我们进一步检查 root age 问题。随后又指出 S-DEC 中也可能存在同类问题。

推进内容包括：

- 补 S-BGB root age/tree scaling。
- 补 S-DEC root age/tree scaling。
- 纠正之前关于“旧版单棵树 DEC”的判断。你指出旧版并没有独立单棵树 DEC，旧版单树 DEC 实际是通过 BioGeoBEARS 六模型路径实现。这一点已纳入后续判断。

## 9. 旧版逻辑调查形成的结论

### 9.1 S-DIVA 与 S-DEC 合并逻辑不完全相同

旧版 S-DIVA 更依赖修改版 DIVA 自己输出统计结果。旧版 S-DEC/BGB 则更接近由外部引擎逐树计算，再按参考树节点或 clade 合并结果。

这说明不能把 S-DIVA 的修复方式简单复制到 S-DEC 或 S-BGB。每个模型都要先确认旧版实际把算法放在哪里。

### 9.2 `/` 和 `*` 的语义

在 BioGeoBEARS/DEC 类结果中：

- `/` 是 null range。
- `*` 是低概率未显示状态的总和，不是祖先分布区。

因此 S-BGB 解析和显示不能把 `*` 当作一个 range，也不能默认去掉 `/`。

### 9.3 Fossil 功能的差异

旧版单树 DIVA 的 Fossil 是给某棵树的某个节点加祖先分布约束。旧版 S-DIVA 中，Fossil 在未勾选 Use Final tree 时不真正约束节点；使用 final tree 时才进入约束逻辑。

Python 版 S-DIVA 已按这个方向调整。

### 9.4 单树 DEC 判断修正

之前曾按“旧版有单树 DEC”讨论。你后来指出旧版其实没有独立单树 DEC，它的单树 DEC 是通过 BioGeoBEARS 六模型实现。这个判断已修正，后续不应再把旧版 DEC 当作一个独立 Lagrange 单树入口。

- 对 DIVA/S-DIVA 做过 proc/batch 写入检查。
- S-BGB 运行速度提升已经通过你的实际使用反馈验证，且原因可以从批处理 runner 架构解释。

没有完成的验证：

- 没有形成自动化回归测试套件。
- 没有对每个模型都固定旧版输出、Python 输出、参数文件并做持续 diff。
- 命令行端到端 DIVA 测试曾被本机 Python 缺少 `ete3` 阻断。

## 12. 真实存在的隐患

以下不是为了凑风险，而是当前项目确实可能留下的问题。

### 12.1 旧版外部引擎可能被导师修改过

S-DIVA 已明确依赖修改版 `DIVA.exe`。S-DEC 的旧版 Lagrange 也很可能是修改版，支持 `includedists/excludedists` 等普通 lagrange-ng 不一定支持的能力。只要外部引擎不一致，完全数值对齐就不能保证。

### 12.2 S-DEC Include/Exclude 仍不是算法入口级等价

Python 版当前使用 lagrange-ng，Include/Exclude 主要是后处理或兼容过滤。旧版如果是在 Lagrange 内部直接限制状态空间，则两者结果可能不同。这个差异目前是有意保留的，因为硬写未知配置风险更大。

### 12.3 BioGeoBEARS 版本差异可能造成 S-BGB 细微偏差

即使都选择 DEC，结果仍可能受以下因素影响：

- BioGeoBEARS 版本。
- R 版本。
- 优化器。
- 初始值。
- include_null_range 默认值。
- display threshold。
- 是否使用旧版 RASP 自带修改脚本。

这类差异不能靠 Python 端强行调百分比解决，应继续从输入脚本和包版本排查。

### 12.4 Random sampling 只对齐行为，不保证旧版随机序列

如果旧版使用特定随机数种子或 VB.NET 随机序列，Python 版即使抽样策略一致，也可能抽到不同树。S 系列百分比可能因此有细微差别。

### 12.5 Threads/Cores 需要大数据压力测试

S-DIVA 和 S-DEC 的 Threads 现在更接近旧版多树并行。S-BGB 是 Cores。并行后速度提升，但外部 exe/R 进程数量、临时文件隔离、内存占用和异常退出仍需要更大数据集测试。

### 12.6 Save/Load setting 不是完整项目文件

旧版 Save/Load setting 本身保存内容也不完整。当前 Python 版主要对齐旧版行为，不应把它理解为完整工程状态保存。尤其是 time-stratified 和 MRCA 的完整可恢复性仍需继续确认。

### 12.7 缺少自动化旧版对照测试

目前很多正确性判断来自你手动用旧版 RASP 跑 Psychotria 数据后反馈。后续如果继续修改，缺少自动化基准会让回归风险变高。

## 13. 当前项目所处位置

当前 Python 版已经不再只是早期功能原型，已经进入“按旧版 RASP 行为复刻”的阶段。

已经明显推进的部分：

- S-DIVA 的算法入口方向已经纠正。
- S-DIVA 配置窗口、DIVA 配置窗口基本形成。
- S-DEC 配置窗口和多数旧版交互逻辑已经补齐。
- S-BGB 配置窗口和批处理运行方式已经明显接近旧版。
- root age/tree scaling 这类会导致系统性结果偏差的问题已经被发现并补入主要路径。

仍然不能说已经完成复刻的部分：

- S-DEC 与旧版修改版 Lagrange 的引擎级差异。
- S-BGB 与旧版 BioGeoBEARS/R 环境的版本差异。
- S-DIVA proc 参数和 Fossil/Final tree 的边界行为仍需要更多旧版案例核对。
- 其他尚未深入复刻的模型和配置窗口还需要按同样方法逐个调查。

## 14. 建议后续推进顺序

建议后续不要直接按“界面缺什么补什么”推进，而是继续保持现在的方法：

1. 对每个模型先确认旧版真实调用链：VB.NET 自己算，还是外部 exe/R 包算。
2. 固定一组测试数据和参数，保存旧版输出作为基准。
3. Python 版先保证输入文件和旧版一致。
4. 再比较外部引擎输出。
5. 最后才修解析和显示。

短期最值得做的事情：

- 为 Psychotria 数据建立 S-DIVA、S-DEC、S-BGB 三套基准结果。
- 保存每次运行的输入配置、外部引擎原始输出和 Python 解析输出。
- 对 S-DEC 向导师确认旧版 Lagrange 是否修改过，以及 `includedists/excludedists` 的确切语义。
- 对 S-BGB 确认旧版 BioGeoBEARS 包版本、R 版本和旧版 RASP 生成的 R 脚本。
- 对 S-DIVA 继续用 Fossil、Use Final tree、Exclude range、Random Step 等组合参数做旧版对照。

## 15. 本阶段结论

本阶段最重要的成果不是某个单独按钮，而是把复刻路线从“Python 自己模拟旧版结果”纠正为“先弄清旧版真实逻辑，再让对应外部引擎承担算法”。这个方向对 S-DIVA 已经产生实质改进，对 S-BGB 解决了明显性能瓶颈，对 S-DEC 则明确了能对齐和不能假装对齐的边界。

接下来如果继续按这个方法推进，项目会更稳；如果回到只看界面和百分比调结果，很容易再次出现表面接近但算法入口不一致的问题。
